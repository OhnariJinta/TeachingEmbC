1.3 純粋関数と副作用

本教材の核心的な概念は 純粋関数 と 副作用 の分離です。

純粋関数とは

同じ入力に対して常に同じ出力を返し、外部の状態を変更しない関数。

/* ✅ 純粋関数: テストが非常に簡単 */
int16_t temperature_convert(uint16_t raw_adc) {
    int32_t mv = (int32_t)raw_adc * 3300 / 4095;
    return (int16_t)(mv / 10);
}

副作用を持つ関数とは

ハードウェアにアクセスしたり、グローバル変数を変更する関数。

/* ⚠️ 副作用あり: HAL経由でハードウェアにアクセス */
void hal_gpio_write(uint8_t pin, uint8_t state);
uint16_t hal_adc_read(uint8_t channel);

第4章: Google Test で純粋関数を検証する

ここから道具の章です。まずは副作用を持たない純粋関数を、Google Test だけでどう検証するかを押さえます。

4.1 extern "C" と名前マングリング

C++ コンパイラは関数名を「名前マングリング」で装飾します（例: add(int, int) → _Z3addii）。extern "C" で「この関数はC言語の命名規則で探す」と指示する必要があります。

❌ これだとリンクエラー 
#include "temperature.h"
✅ extern "C" で解決 
extern "C" {
#include "temperature.h"
}

4.2 TEST マクロ（最もシンプル）

#include "gtest/gtest.h"

extern "C" {
#include "temperature.h"
}

TEST(TemperatureConvert, ZeroInput) {
    EXPECT_EQ(0, temperature_convert(0));
}
  • 第1引数: テストスイート名（機能や関数の名前）
  • 第2引数: テストケース名（テストの目的）

4.3 TEST_F マクロ（フィクスチャ付き）

class TemperatureConvertTest : public ::testing::Test {

4.4 主要なアサーションマクロ

マクロ意味失敗時の動作
EXPECT_EQ(expected, actual)等しいテスト続行
EXPECT_NE(val1, val2)等しくないテスト続行
EXPECT_GT(val1, val2)val1 > val2テスト続行
EXPECT_GE(val1, val2)val1 >= val2テスト続行
EXPECT_LT(val1, val2)val1 < val2テスト続行
EXPECT_LE(val1, val2)val1 <= val2テスト続行
EXPECT_TRUE(condition)真テスト続行
EXPECT_FALSE(condition)偽テスト続行
ASSERT_EQ(expected, actual)等しいテスト中断
EXPECT vs ASSERT: EXPECT_* は失敗してもテストを続行し、全ての失敗を報告します。ASSERT_* は失敗した時点でテストを中断します。

4.5 実例: 純粋関数のテスト

// test_app.cpp — 純粋関数テスト（フェイク不要）
#include "gtest/gtest.h"

extern "C" {
#include "temperature.h"
}

class TemperatureConvertTest : public ::testing::Test {};

TEST_F(TemperatureConvertTest, ZeroRawReturnsZero) {
    EXPECT_EQ(0, temperature_convert(0));
}

TEST_F(TemperatureConvertTest, MidRangeValue) {
    int16_t result = temperature_convert(2048);
    EXPECT_GE(result, 160);
    EXPECT_LE(result, 170);
}

TEST_F(TemperatureConvertTest, MaxRawValue) {
    EXPECT_EQ(330, temperature_convert(4095));
}
ポイント: 純粋関数は入力と出力だけを検証すればよいため、フェイクもモックも不要。これが「テスト容易性」の好例です。

純粋関数としての状態遷移

temp_alarm_fsm_t temp_alarm_transition(
    const temp_alarm_fsm_t *current,
    const temp_alarm_event_t *event) {
    temp_alarm_fsm_t next = *current;

    switch (event->type) {
    case TEMP_ALARM_EVENT_ADC_READY:
        next = temp_alarm_apply_sample(&next, event->raw_adc);
        break;
    case TEMP_ALARM_EVENT_ACK_BUTTON:
        next.state = TEMP_ALARM_STATE_MONITORING;
        next.alarm_led_on = 0;
        break;
    default:
        break;
    }

    return next;
}
純粋関数化の利点: 入力は「現在状態 + イベント」、出力は「次状態」です。外部I/Oを触らないため、同じ入力なら同じ結果になり、Google Test だけで検証できます。

  • HAL の具体実装に引きずられる
✅ After
  • SRP: temp_alarm_transition() は遷移だけを担当する
  • DIP: 状態機械は temp_alarm_event_t という抽象イベントに依存する
  • OCP: 新イベントは enum と switch の拡張で追加できる

第9章: ISR設計と副作用の分離

第8章で作った状態機械を、実際の割り込み入口へどう接続するかを扱う章です。ISR は短く、決定論的で、責務が小さいほど扱いやすくなります。ここでは悪い ISR の具体例から始めて、イベント発行だけに絞った ISR へリファクタリングし、純粋関数のロジックと副作用の境界をどう守るかまで整理します。

9.1 Before: ISR の中で全部やってしまう

❌ Before: bad_isr.c

9.2 After: ISR を薄くし、ロジックは外へ出す

sequenceDiagram participant TimerISR as Timer ISR participant AdcISR as ADC ISR participant AckISR as ACK ISR participant FSM as temp_alarm_fsm_dispatch() participant Pure as temp_alarm_transition() TimerISR->>FSM: TIMER_TICK AdcISR->>FSM: ADC_READY(raw_adc) AckISR->>FSM: ACK_BUTTON FSM->>Pure: current + event Pure->>FSM: next state
ISR役割イベント
タイマ割り込み周期サンプリングの要求を上げるTIMER_TICK
ADC完了割り込み取得済みのADC値を状態機械へ渡すADC_READY
ACKボタン割り込みユーザの確認入力を通知するACK_BUTTON
ロジックは純粋関数に、ハードウェアアクセスはHAL関数に分離する。イベント境界を揃えると、ISR でも通常関数でも同じ仕様で扱えます。

ISRの実装例

    temp_alarm_fsm_dispatch(fsm, &event);
}

void temp_alarm_fsm_on_adc_interrupt(temp_alarm_fsm_t *fsm, uint16_t raw_adc) {
    const temp_alarm_event_t event = { TEMP_ALARM_EVENT_ADC_READY, raw_adc };
    temp_alarm_fsm_dispatch(fsm, &event);
}

void temp_alarm_fsm_on_ack_interrupt(temp_alarm_fsm_t *fsm) {
    const temp_alarm_event_t event = { TEMP_ALARM_EVENT_ACK_BUTTON, 0 };
    temp_alarm_fsm_dispatch(fsm, &event);

ISRでやらないこと: 長いループ、複雑な分岐、フォーマット処理、ブロッキング待ち、複数モジュールへの直接書き込みは避ける。ISRが肥大化すると、応答性が落ち、ホストテストもしづらくなります。

9.3 SOLID / DIP の観点

❌ Before
  • SRP違反: ISR が I/O、判定、ログ出力まで担当する
  • DIP違反: 割り込み入口が HAL や UART 具体実装へ直接依存する
  • ホストテストしにくく、実機依存が強い
✅ After
  • SRP: ISR はイベントを作るだけにする
  • DIP: ISR は temp_alarm_event_t と temp_alarm_fsm_dispatch() に依存する
  • 副作用は別レイヤへ押し出せる
